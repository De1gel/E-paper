# 刷新策略笔记（全刷 / 快刷 / 局刷）

更新时间：2026-04-01

## 1. 当前工程已具备能力
- 全刷初始化链：`EPD_init()`
- 快速初始化链：`EPD_init_fast()`
- 全屏数据写入：`DTM (0x10)` + `DRF (0x12)`
- 休眠链：`POF/DSLP` 路径（工程中通过 `EPD_sleep()` 使用）

对应代码：
- `src/Display_EPD_W21.cpp`
- `src/app/App.cpp`

## 2. 已验证的局刷相关事实
- 从“局刷资料包”中确认了窗口命令链：
  - `CMD_PARTIAL_WINDOW = 0x83`
  - 后续写入窗口坐标（x/x_end/y/y_end）
  - 局部写入 `DTM` 数据，再触发刷新相关命令
- 本项目已将该思路抽象为模块：
  - `src/display/PartialRefresh.h`
  - `src/display/PartialRefresh.cpp`

## 3. 刷新模式的工程定位
- **全刷**：稳定基线，用于页面切换、大范围变更、残影清理。
- **快刷**：在可接受闪烁条件下提升速度，适合常规轮播页更新。
- **局刷（实验态）**：用于小区域更新验证；是否“其他区域不闪”取决于控制器波形和时序匹配。

## 4. 风险与边界
- 仅有窗口命令不代表可长期稳定局刷。
- 若缺乏匹配 LUT/波形资料，可能出现：
  - 残影累积
  - 局部闪烁或整屏联动闪
  - 颜色漂移

## 5. 推荐落地路径
1. 先把刷新策略抽象为统一接口（Full/Fast/PartialExperiment）。
2. 主流程默认仍走 Full/Fast，Partial 仅在实验开关下启用。
3. 给 Partial 增加长稳测试指标：
  - N 次局刷后残影等级
  - 刷新时长统计
  - 闪烁主观等级
4. 满足阈值后再考虑纳入产品主流程。
