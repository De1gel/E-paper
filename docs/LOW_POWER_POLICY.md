# 低功耗策略与当前实现状态

更新时间：2026-04-10  
依据：`D:\MyCode\ESP\设计思路资料\低功耗思路.txt`

## 1. 目标策略

项目目标仍然是最低功耗路径：
- 刷新完成后让 EPD 进入 `sleep`
- 通过 `IO32` 切断外设电源
- 在非交互时段让 ESP32 本体进入 `light/deep sleep`

## 2. 当前代码已落实的部分

### 渲染会话

当前 `App` 已实现：
- 渲染前打开外设电源
- 调用 `EPD_init_fast()`
- 渲染后调用 `EPD_sleep()`
- 再把 `IO32` 拉低

同时：
- 关闭外设电源前会卸载 SD
- 照片页与日历页共用这条显示会话路径

### WiFi Portal 会话

当前 `WifiManager` 已实现：
- 进入 AP / STA 前拉高 `IO32`
- 停止 WiFi 会话后拉低 `IO32`

说明：
- 这是因为 Portal 期间需要访问 SD、传感器和静态资源
- 因此 `IO32 LOW` 不会覆盖整个 WiFi 会话周期

### 模式超时

当前代码中已经落地：
- `ConfigWait` 超时：`60s`
- AP 空闲超时：`300s`
- STA 连接超时：`30s`

## 3. 当前代码还没有落实的部分

- 没有 `esp_light_sleep_*` / `esp_deep_sleep_*` 调用
- `loop()` 仍是固定轮询 + `delay(50)`
- `Normal` 模式不会自动进入 MCU 级休眠
- STA 运行态空闲超时当前关闭（`kStaSessionTimeoutMs = 0`）

结论：
- 当前已经做到“EPD 低功耗 + 外设电源收敛”
- 还没有做到“整机级低功耗状态机”

## 4. 继续开发时的硬约束

- 不要绕过 `IO32` 的统一外设电源控制
- 不要在未刷新时长时间保持 EPD 供电
- 不要新增长期 busy wait
- 不要让新增配置模式绕过现有超时机制

## 5. 当前验收判断

### 已满足

- 屏幕刷新后，应用层会关闭外设电源
- 配置等待模式具备自动退出
- WiFi 进入 / 退出会同步控制外设电源

### 未满足

- 轮播间隔期间 MCU 仍处于普通轮询状态
- 尚未形成按模式分级的整机休眠策略

## 6. 下一步建议

1. 在 `Normal` 模式引入 MCU 级休眠  
   - 让照片轮播等待期和日历等待期不再空转

2. 明确 `ConfigAP / ConfigSTA` 会话的功耗边界  
   - 特别是 STA 运行态是否需要恢复空闲超时

3. 把低功耗验收拆成两层  
   - 面板 / 外设电源收敛
   - MCU 级休眠收敛
